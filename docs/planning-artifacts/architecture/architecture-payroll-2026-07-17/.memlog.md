- (decision) PARADIGM: Functional core, imperative shell (a.k.a. hexagonal-lite). Pure domain module owns all fairness math; adapters (HTTP/RSC, Prisma, filesystem, clock, PRNG) live in the shell and depend inward only. Binds: all. Prevents: fairness math leaking into SQL, React components, or route handlers where it becomes untestable and forks. Chosen because the SPEC already demands what this paradigm gives for free: answers are pure functions of (data, as-of date, threshold), tests fast+deterministic with no wall clock. Alternatives weighed: (a) layered CRUD + service classes -- rejected, invites math in services that reach for the DB and the clock; (b) DDD aggregates + event sourcing -- rejected, the append-only salary series is already the only event log needed; full ES is ceremony for one user and 10k rows.
- (decision) AD: All fairness math (median, spread, distance %, outlier flag, gender gap, overdue) is computed in-process in pure TypeScript over an in-memory set. SQL never computes a statistic -- no percentile_cont, no AVG, no window functions for domain values. Prevents: the single sharpest divergence -- SQL percentile_cont interpolates continuously while a hand-written TS median picks/averages discretely, so two independently-built units answering the same CAP-5 question return DIFFERENT medians. Also keeps CAP-11's determinism testable without a database. Cost accepted: full-set load per sweep; 10k rows is trivial.
- (decision) AD: Median is defined exactly once, canonically: sort ascending by integer minor units; odd n -> middle element; even n -> arithmetic mean of the two middle elements, rounded half-up to the nearest minor unit. Prevents: lower-median vs upper-median vs interpolated-median divergence between the peer-comparison view (CAP-5), the outlier sweep (CAP-6), and the gender-gap view (CAP-7) -- three consumers of one number.
- (decision) AD: Money is represented as integer minor units + ISO-4217 currency code, never a float and never a bare number. Minor-unit exponent comes from the currency reference table (INR/EUR=2, JPY=0). Prevents: float drift making medians and distance percentages irreproducible, and violates the boundary-exactness CAP-6 demands; also prevents one unit assuming 2 decimals everywhere and mis-rendering JPY.
- (decision) AD: Distance from peer median is computed as (salary - median)/median, rounded half-up to ONE decimal place, and the outlier flag tests that ROUNDED value strictly greater than threshold. Prevents: (a) a badge reading '20.0% above median' while the row is flagged (unrounded 20.04) -- displayed math must reproduce by hand; (b) the undefined boundary at exactly 20.0 (SPEC fixes 19.9 no / 20.1 yes but is silent at 20.0) -- strict > means 20.0 does not flag.
- (decision) AD: Currency is stored ON each salary record, denormalized from the employee's country at write time via the country reference table, and validated to equal it. Settles EXPERIENCE.md Note 2. Prevents: resolving currency from employee.country retroactively rewriting the currency of immutable historical records if country ever changes -- an append-only series must carry its own currency. The write-time check preserves the SPEC's structural guarantee that no comparison crosses a currency.
- (decision) AD: Import validates country against the reference table and rejects unknown countries per-row, same as role and level. Settles EXPERIENCE.md Note 4 (where SPEC mandated rejection only for role/level). Prevents: an unknown country silently creating a peer group of one AND having no currency, breaking both the n>=5 refusal semantics and currency isolation. Country is part of peer identity and determines currency, so it is load-bearing in exactly the way role and level are.
- (decision) AD: Same-effective_from tie-break is by monotonically increasing insertion sequence (surrogate BIGSERIAL id); the highest id wins. Settles EXPERIENCE.md Note 5 / Flow 3's correction case. Prevents: two records dated the same day resolving differently per query; created_at rejected as tie-break because it reads the wall clock, which the determinism promise forbids.
- (decision) AD: Spread is min-max of the peer group's as-of current salaries in the group's single currency. Settles EXPERIENCE.md Note 1. Prevents: IQR/percentile divergence; adopts the SPEC's own stated reasoning for rejecting quartiles (unreliable at n=5-10) and matches the UX display decision, so stored measure and shown measure cannot fork.
- (decision) AD: Employee identity is an opaque surrogate id (UUIDv7), generated in the shell, never derived from name. Settles EXPERIENCE.md Note 3. Prevents: name-as-key collisions across 10,000 people and unstable URLs when a name is corrected. UUIDv7 over UUIDv4 for index locality; over BIGSERIAL because ids appear in URLs and enumerable ids leak headcount.
- (decision) AD: The as-of date is a required explicit parameter on every domain function; no domain code reads the system clock or Date.now(). The clock is a shell adapter that supplies 'today' only as a DEFAULT at the HTTP/RSC boundary. Prevents: the determinism promise dying quietly -- one unit defaulting as-of internally makes the same question answer differently tomorrow and makes tests wall-clock-dependent, which the SPEC forbids outright.
- (decision) AD: Findings (CAP-6 sweep) are computed fresh per request from the as-of current-salary set. No materialized outlier table, no cache, no seen/unseen state. Prevents: a cache going stale against a changed salary and re-introducing the exact untrustworthiness the product exists to kill; also enforces EXPERIENCE.md's 'pure function of data + threshold + as-of date' findings-list contract. Cost accepted: ~10k-row read per sweep.
- (decision) AD: FX rates live in a pinned rate table keyed (from, to, pinned_on). Org-wide totals spanning currencies use the latest rate set with pinned_on on or before the as-of date, and carry that date in the response payload -- not just the UI. Prevents: a converted figure being displayed without its receipts because the rate was resolved in the view layer; keeps CAP-9 deterministic under a wound-back as-of date.
- (decision) AD: Seed (CAP-11) draws from an injected seeded PRNG; Math.random is banned repo-wide (lint). Log-normal via Box-Muller over the seeded stream. Prevents: an irreproducible 10k population, which would break CAP-11's 'fixed seed, reproducibly' success criterion and make planted outliers/thin cells/gender effects unverifiable.
- (decision) STACK: Next.js 16.2.10 App Router as one full-stack deployable; React Server Components call the domain through server-only query modules; Route Handlers only where a non-RSC surface needs them (CSV export, import upload). Alternatives weighed: separate Fastify/Express API + React SPA -- rejected: two deployables, CORS, duplicated types, and a network hop for a single-user 10k-row tool that buys nothing. Next.js satisfies the JD's Node/TS + React requirement in one surface.
- (decision) STACK: PostgreSQL (Neon serverless) over SQLite. SPEC/THA permits SQLite explicitly, but 'deployed and demonstrable' on Vercel is serverless with an ephemeral filesystem, where SQLite does not survive. Alternatives weighed: (a) SQLite + Fly.io/Railway volume -- viable, rejected for a heavier deploy story; (b) Turso/libSQL -- rejected as the less boring option. Note the paradigm makes this cheap to reverse: no domain code touches SQL, so the DB is swappable behind the repository port.
- (decision) STACK: Prisma 7.8.0 as the ORM.
