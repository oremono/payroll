Where architecture starts

The product questions were already settled

The SPEC arrived having decided what most projects argue about for a month.

peerA peer group is role + level + country. Nothing else defines one.
historySalary is append-only and effective-dated. Nothing is ever overwritten.
outlierUnderpaid and overpaid are the same finding. Symmetric, direction-agnostic.
fxNo comparison crosses a currency. Conversion exists only in aggregate totals, pinned and dated.
refusalBelow n=5, refuse — never widen the group.
as-ofEvery answer is a function of the data and a supplied as-of date. Never the clock.

So the spine's job was never "what should this do." It was one question, asked twenty-four times:

If two engineers built different features from this contract independently — could they choose incompatibly? Fix it only where the answer is yes, the call is non-obvious, and it is a real trade-off. Everything else is deferred and named.

Rejected

Layered CRUD + services

A service that can reach the repository will reach it mid-calculation. A service that can read the clock will default the as-of date "just this once." Both are invisible in review. Both are fatal.

Rejected

DDD + event sourcing

The append-only salary series is an event log — and it's the only one needed. Projections and replay for one user and 10,000 rows is ceremony without a payer.

AD-1 Enforced by an import-boundary lint rule in CI — because the failure it prevents compiles cleanly and passes review.

The decision that shaped everything else

SQL computes nothing

The database stores rows and selects sets. It computes no median, spread, distance, gap, count, or total that reaches a user.

Why this is the sharpest divergence in the product

Postgres percentile_cont interpolates continuously. A hand-written TypeScript median picks or averages discretely. Both are correct implementations of "the median."

Two engineers — one building the peer-comparison card, one building the outlier sweep — ship different answers to the same question. Both pass review. Nothing announces the disagreement. Alice pastes one of them into Slack.

The follow-through

One canonical median

Sort by integer minor units. Odd n → the middle. Even n → the mean of the two middle values, rounded half-up. One implementation exists. No other median may be written.

AD-3

The cost, accepted

Full sets into memory

We give up the database's optimised statistics and load ~10,000 rows per sweep. At this size that's nothing — and correctness that unit-tests in milliseconds beats a query plan.

reversible no domain code touches SQL, so the database stays swappable.

What the review gate found — in this spine's own first draft

The bug that would have shipped

AD-5, as originally written: distance is (salary − median) / median, and the outlier flag tests that value with a strict > against the threshold.

The defect

−25.2 > 20 is false.

A literal implementation never flags an underpaid employee. Half of CAP-6 disappears. Half of the seeded planted outliers become undetectable. The product's loudest promise — being underpaid and being overpaid are the same finding — inverts into its opposite.

And the demo still looks fine, because the above-median outliers still show.

Corrected rule
  • distance in percentage points
  • exact decimal arithmetic — in an IEEE double, 20.05 is 20.049999… and never rounds up to flag
  • round the magnitude, then reapply the sign — so +20.05 and −20.05 round symmetrically
  • the flag tests |d| > threshold, strictly
Two calls that rode along

The number shown is the number judged

The flag tests the rounded value — so a badge can never read 20.0% above median on a row that flagged at an unrounded 20.04.

Exactly 20.0 does not flag

The SPEC fixes 19.9 → no and 20.1 → yes, and is silent between. Strict > settles it.

Three independent reviewers found this. None of them found it by rereading.

Five questions UX refused to invent answers to

Handed to architecture, and settled

Spread measureAD-9 min–maxThe SPEC already rejected quartiles and standard deviations as unreliable at n = 5–10. The same reasoning kills IQR — and min–max matches what the UI shows, so stored and displayed cannot fork.

Holes only an adversary found

Obey every rule. Still build incompatibly.

A reviewer was tasked with constructing pairs of units that follow every AD to the letter and still disagree. It produced eighteen. Four became new invariants.

Collapsed six findings into one

AD-16 The as-of population was undefined

Nothing said who is in a peer group at a date. One obedient unit counts everyone matching the triple — n=6, so it answers. Another counts only those with a salary as of that date — n=4, so it refuses. The same group, the same date, simultaneously answered and refused — and the card prints "Based on 6 peers" above a median computed from 4. Exactly the failure the n≥5 refusal exists to prevent, reintroduced by an unstated definition.

AD-17 No gap formula

Five defensible readings gave 8%, −8.7%, or ₹2,00,000 from identical data — in the sentence Alice pastes into Slack.

AD-18 Append-only unenforced

The SPEC's loudest constraint was held by discipline. update() would compile and pass every test — while purity, determinism, and randomness each had a gate.

AD-22 Overdue read the clock

Nothing bound CAP-10's cutoff to the as-of date, so winding the date back would not reproduce yesterday's list.

Also caught: a regression introduced during the fixes — the clock ban narrowed from domain + application to domain alone, quietly re-permitting new Date() inside use-cases.

The other half of the contract

What was deliberately not decided

A spine stays lean by naming what it won't settle, and why it can wait.

Authentication

A SPEC non-goal — and the only deferral that must flip before this touches a real salary record. Deferred, not dismissed. The trigger is explicit: the moment the HR team rather than the manager alone is a user.

Employee country change

A recorded deviation, not a clean deferral — the SPEC names country among an edited employee's fields; this spine makes it immutable after create. The prohibition is binding in AD-6, not filed under Deferred, because it's the only thing preventing a mixed-currency peer group.

Caching / read models

Findings are computed fresh, every request. A cache going stale against a changed salary would reintroduce the exact untrustworthiness this product exists to kill.

Observability, backups, rate limiting

No operational stakes at one user with a population regenerable from one command. Revisit the moment this holds data that isn't reproducible.

Note the shape of these. Each names a revisit condition rather than a date. A deferral without a trigger is just an omission with better manners.
The build discipline, not just the structure

Test-first, feature by feature

The SPEC asks for tests that are fast, deterministic, and honest. Two decisions govern how the code arrives — and the assessment reads both test quality and commit history.

AD-23 Tests are written first

Red → green → refactor. A failing test precedes the code that satisfies it — the cheapest way to keep a test describing the behaviour that was wanted, not the code that happens to exist.

The honest split: CI can't prove test-first, so ordering is enforced in review — the commits have to show it. CI enforces what it can: a coverage floor, and mutation testing over the domain — a surviving mutant is a line no test actually pins down.

AD-24 Vertical slices, backend first

One capability at a time; within it, the backend story lands before the frontend. A horizontal build would let the AD-20 answer payload be designed twice — once by the backend, once differently by the component that consumes it — the drift AD-20 exists to kill, smuggled back through build order.

"Backend done" is a gate, not a mood: domain + application green, one adapter integration test against a real disposable Postgres 18 (never a mock), and the AD-20 payload finalized.

The first slice isn't a capability at all. It's the foundational app shell — the import-boundary lint (AD-1), the token build (AD-15), the two route handlers (AD-21) — so every capability slice afterward lands on rails that are already standing.
How this was built

The gate is the method, not the formality

The spine was drafted fast, then attacked.

lintDeterministic pass — placeholders, duplicate IDs, missing Binds/Prevents/Rule, unpinned versions.
×5Parallel independent lenses: reconcile against the SPEC · reconcile against both UX spines · rubric walk · web re-verification of every version · adversarial hunt for obedient-but-incompatible unit pairs.
×3Rounds. Each verified the prior round's fixes and hunted for what the fixes broke.

What it caught, in the author's own draft

  • An inverted outlier test that would have silently deleted every underpaid finding
  • An undefined population letting two capabilities answer and refuse the same group
  • An unowned gender-gap formula with five defensible readings
  • An unenforced append-only guarantee — the SPEC's loudest constraint
  • A stale Postgres pin, on the one version flagged as unverified
  • A regression introduced during the fixes

None of these were found by rereading. All were found by independent readers with a specific adversarial brief. That's the reusable lesson — and the reason the gate runs before handoff rather than after.
