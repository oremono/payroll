- (decision) CAP-6 OUTLIER THRESHOLD RESOLVED (was the open question). Method: PERCENTAGE distance from peer median — not SD, IQR/Tukey, or percentile rank. Rationale: peer groups are small by construction (n>=5, typically <10) and every distribution-fitting method is unreliable at that size — an SD from 5 points is noise, IQR quartiles are 2 data points, 'bottom 10%' of 8 people is <1 person. A percentage is stable at any n and explainable to the user in her own words.
- (decision) rk: underpaid and overpaid are the SAME FINDING — outlier detection is symmetric/direction-agnostic.
- (decision) rk chose CONFIGURABLE threshold. Default 20%, user-adjustable. DEFAULT GROUNDING: the attested 80-120% compa-ratio corridor is the industry's own tolerance for distance from a reference point; peer median plays the structural role a band midpoint would, so +/-20% is borrowed from the domain rather than invented — the same move as substituting a derived statistic for a benchmark ACME does not own. Testing benefit: threshold becomes a parameter, so outlier detection is deterministic given (data, threshold) and the boundary is exactly testable (19.9% no, 20.1% yes).
- (note) Expect the outlier list to lean HIGH: the log-normal seed has a long upper tail by construction, so a symmetric % rule flags more above the median than below. That is the distribution showing through, not ACME overpaying. Relevant when reading the demo.
