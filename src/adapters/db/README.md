# `src/adapters/db/`

Prisma client + repository implementations (implementing `application/ports/`). Populated from
Story **1-3** (data model & migrations).

Import rule (inherited from the adapters layer): **`application`, `domain`**.

- `client.ts` — the single Prisma client, `globalThis`-guarded against dev hot-reload
  multiplication. Construction is **lazy** (`getDbClient()`), so importing this module never opens
  a connection: the `check` and `a11y` CI jobs build and serve the app with no database at all.
  The `@prisma/adapter-pg` driver adapter is built **inside** the guard — Prisma's own documented
  snippet builds it outside, which leaks a `pg` pool on every HMR reload.
- `generated/` — the Prisma client, generated by `prisma generate` (wired to `postinstall`).
  Git-ignored, and excluded from ESLint, coverage, Stryker, and `tsc`. **Never edit by hand.**

Prisma lives **only** in this directory (Law 2). `src/domain/**` and `src/application/**` may not
import `@prisma/client`, `@prisma/adapter-pg`, or the generated path — `eslint.config.mjs` enforces
all three. Under Prisma 7 a leak names the *generated path*, not `@prisma/client`, which is why
the purity rule bans that too.

Repositories land with their first consumer (CAP-2 / CAP-3), not here: Story 1-3 proves
append-only at the database, and the typed port arrives when a use-case needs it.
